# Multi-Developer Hub Ref Conflict Scenarios

When multiple developers push to the hub simultaneously, git's standard merge
machinery interacts with submodule ref pointers in ways that can silently lose
work. This document catalogues four concrete conflict scenarios, their
resolution paths, and data loss risk.

**Related:** [workflow-pain-points.md](workflow-pain-points.md) (pain points #1, #7),
[submodule-drift.md](submodule-drift.md), [hub-workflow-audit.md](hub-workflow-audit.md)

---

## Scenario 1: Different Submodule Refs — Push Conflict on Hub

### Setup

- Dev A updates `projects/billing` (advances by 3 commits)
- Dev B updates `projects/auth` (advances by 2 commits)
- Both started from the same hub commit

### Sequence

```
Time  Dev A                                Dev B
───── ──────────────────────────────────── ──────────────────────────────────────
 T1   cd projects/billing                  cd projects/auth
      git add . && git commit && git push  git add . && git commit && git push
 T2   cd hub-root                          cd hub-root
      git add projects/billing             git add projects/auth
      git add .project/                    git add .project/
      git commit -m "billing: feature X"   git commit -m "auth: fix Y"
 T3   git push   ← succeeds               git push   ← REJECTED (non-fast-forward)
```

### Conflict Description

Git rejects Dev B's push because the hub's `main` has moved forward (Dev A's
commit). Both commits modify the hub's tree, but they touch **different**
submodule refs (`projects/billing` vs `projects/auth`).

### Can Git Auto-Resolve?

**Yes.** Dev B runs `git pull --rebase` (or `git pull`). Git sees that:
- Dev A changed `projects/billing` (submodule ref)
- Dev B changed `projects/auth` (submodule ref)

These are changes to different paths, so git merges (or rebases) cleanly. Both
submodule refs survive in the merged result.

**Exception:** If both developers also modified `.project/` files (e.g., both
updated `index.yaml` or the same story file), those paths **will** conflict and
require manual merge resolution.

### Recovery Path

```bash
# Dev B:
git pull --rebase    # Rebases Dev B's commit on top of Dev A's
git push             # Now succeeds — both refs present
```

### Data Loss Risk: NONE (for submodule refs)

The submodule ref changes are in different tree entries. Git's merge handles
this correctly. The only risk is `.project/` file conflicts if both developers
modified PM data in the same files (e.g., both updated `index.yaml` — which
is auto-generated and will conflict on every concurrent update).

### `.project/index.yaml` — The Hidden Conflict

Even when developers touch completely different submodules,
`server.py` rebuilds `.project/index.yaml` after every PM mutation. If both
developers performed PM operations (e.g., `pm_update`, `pm_create_story`),
their `index.yaml` files will diverge. This file is auto-generated and contains
running totals, so merge conflicts in it are:

- **Guaranteed** whenever two developers do PM operations concurrently
- **Annoying** because the file is machine-generated and should not be
  hand-merged
- **Low risk** because `index.yaml` can be regenerated by running
  `pm_reindex`, but the developer must know to do this

## Scenario 2: Same Submodule Ref — Which Commit Wins?

### Setup

- Dev A and Dev B both update `projects/billing`
- Both started from the same hub commit (billing at commit `aaa111`)
- Dev A advances billing to commit `bbb222` (5 new commits on `main`)
- Dev B advances billing to commit `ccc333` (3 different commits, branched
  from the same starting point `aaa111`)

### Sequence

```
Time  Dev A                                Dev B
───── ──────────────────────────────────── ──────────────────────────────────────
 T1   cd projects/billing                  cd projects/billing
      # 5 commits on main                  # 3 commits on feature/payments
      git push                             git push -u origin feature/payments
 T2   cd hub-root                          cd hub-root
      git add projects/billing             git add projects/billing
      git commit -m "billing: new API"     git commit -m "billing: payments v2"
 T3   git push   ← succeeds               git push   ← REJECTED
```

### Conflict Description

Dev B pulls the hub: `git pull --rebase`. Git attempts to merge the tree. Both
commits modified the **same path** (`projects/billing`), changing the submodule
ref from `aaa111` to different values (`bbb222` vs `ccc333`).

### Can Git Auto-Resolve?

**No.** This is a genuine merge conflict. Git cannot choose between two
different submodule commit SHAs for the same path. The developer sees:

```
CONFLICT (submodule): Merge conflict in projects/billing
Automatic merge failed; fix conflicts then commit the result.
```

### Recovery Path

Dev B must decide which submodule commit the hub should reference:

```bash
# Option A: Keep Dev A's ref (most common — accept what's on remote)
git checkout --theirs projects/billing
git add projects/billing
git rebase --continue

# Option B: Keep Dev B's ref (requires team coordination)
git checkout --ours projects/billing
git add projects/billing
git rebase --continue

# Option C: Merge the submodule branches first, then use the merged commit
cd projects/billing
git checkout main                    # Dev A's branch
git merge feature/payments           # Incorporate Dev B's work
git push
cd hub-root
git add projects/billing             # Now points at the merge commit
git rebase --continue
```

### Data Loss Risk: MEDIUM

- **No commits are lost** in the submodule — both `bbb222` and `ccc333` exist
  in the billing remote
- **The hub ref can only point at one commit.** Whichever ref the resolver
  doesn't pick becomes unreachable from the hub. A developer must know to merge
  the submodule branches to preserve both lines of work
- **If Dev B chooses Option A or B without merging:** one developer's submodule
  work is invisible from the hub until someone updates the ref again
- **If Dev B doesn't understand submodule conflicts:** they may resolve the
  conflict incorrectly (e.g., by deleting the submodule entry or pointing it at
  a wrong commit)

## Scenario 3: Stale Hub — Submodule Refs Revert to Old Commits

### Setup

- Dev A pushes the hub, advancing `projects/api` from commit `aaa` to `bbb`
- Dev B has not pulled the hub — their local hub still references `aaa` for
  `projects/api`
- Dev B is working on `projects/frontend` (unrelated submodule)

### Sequence

```
Time  Dev A                                Dev B
───── ──────────────────────────────────── ──────────────────────────────────────
 T1   # Advances projects/api              # Working on projects/frontend
      cd projects/api                      cd projects/frontend
      git add . && git commit && git push  git add . && git commit && git push
 T2   cd hub-root                          (hasn't pulled hub)
      git add projects/api .project/
      git commit && git push  ← succeeds
 T3                                        cd hub-root
                                           git add projects/frontend .project/
                                           git commit -m "frontend: update"
                                           git push  ← REJECTED
 T4                                        git pull --rebase
                                           # Rebase succeeds — different paths
                                           git push  ← succeeds
```

### Conflict Description

At T4, `git pull --rebase` merges cleanly because Dev A changed
`projects/api` and Dev B changed `projects/frontend` — different paths.
**But there is a subtle problem:** Dev B's commit does not include
`projects/api` in its changeset. After rebasing on top of Dev A's commit, the
hub correctly shows:

- `projects/api` → `bbb` (from Dev A's commit)
- `projects/frontend` → Dev B's new commit

**This scenario is actually safe** when Dev B only stages the files they
changed. Git's rebase preserves Dev A's changes to other paths.

### When It Goes Wrong

The dangerous variant occurs when Dev B runs `git add -A` or `git add .`
instead of staging specific files:

```
Time  Dev B (dangerous variant)
───── ──────────────────────────────────────────────────────────────────
 T3   cd hub-root
      git add -A                         # Stages EVERYTHING including
                                         # projects/api (still at old ref aaa)
      git commit -m "frontend: update"
      git push  ← REJECTED
 T4   git pull --rebase
      # Now git sees BOTH commits modified projects/api:
      #   Dev A: aaa → bbb
      #   Dev B: aaa → aaa (explicit stage of the old ref)
      # This is a CONFLICT on projects/api
```

If Dev B used `git add -A`, they explicitly staged the old submodule ref for
`projects/api`. This creates a conflict with Dev A's update. If Dev B resolves
by keeping "theirs" (Dev A's ref), no harm done. But if Dev B resolves by
keeping "ours" (their stale ref), **Dev A's submodule advance is reverted**.

### Can Git Auto-Resolve?

- **Selective staging (`git add projects/frontend`):** Yes, clean merge. Safe.
- **Blanket staging (`git add -A`):** No. Conflict on `projects/api` because
  both commits touch that path.

### Recovery Path (if ref was reverted)

1. Someone notices `projects/api` points at `aaa` instead of `bbb`
   (no automated detection exists)
2. Pull the hub, advance the ref:
   ```bash
   cd projects/api
   git checkout main && git pull
   cd hub-root
   git add projects/api
   git commit -m "restore api ref to latest"
   git push
   ```
3. Notify the team to pull the hub

### Data Loss Risk: HIGH (with `git add -A`)

- Submodule ref silently reverted to an older commit
- Dev A's work still exists in the submodule remote but is unreachable from
  the hub
- No warning or error at any point — the revert looks like a normal commit
- Difficult to detect: the hub commit message says "frontend: update" but
  also changed the api ref, and `git log` doesn't highlight submodule ref
  changes prominently

**With selective staging:** NONE. This is why `git add -A` in a hub repo is
dangerous — it silently stages stale submodule refs.

## Scenario 4: Nightly Cron Sync During Developer Mid-Push

### Setup

- A cron job runs nightly at 02:00: `projectman sync && projectman audit`
  (see [cron.md](cron.md))
- Dev A is working late and is in the middle of a push sequence

### Sequence

```
Time   Dev A (manual)                     Cron (automated)
────── ──────────────────────────────────  ──────────────────────────────────
01:58  cd projects/api
       git add . && git commit && git push
       cd hub-root
       git add projects/api
01:59  git commit -m "api: late fix"
       git push   ← succeeds
                                           ┌─────────────────────────────────
02:00                                      │ projectman sync
                                           │ → git pull --ff-only per submodule
                                           │ → projects/api: updated (pulls
                                           │   Dev A's latest push)
                                           │ → other submodules: updated
                                           │ projectman audit
                                           │ → rebuilds indexes, dashboards
                                           │ → writes DRIFT.md, index.yaml
                                           └─────────────────────────────────
02:01  # Dev A continues working...
       # Makes more changes to projects/api
       cd projects/api
       git add . && git commit && git push
       cd hub-root
       git add projects/api .project/
       git commit -m "api: follow-up"
       git push   ← succeeds
```

### Conflict Description

**The cron job does NOT commit or push.** `projectman sync` only runs
`git pull --ff-only` inside each submodule directory (`registry.py:463-464`).
It does not stage, commit, or push the hub. Similarly, `projectman audit`
writes files to `.project/` but does not commit them.

This means:

1. The cron job leaves dirty files in the hub's working tree (updated
   `DRIFT.md`, `index.yaml`, dashboards)
2. **If Dev A runs `git add -A`** after the cron: they will inadvertently
   stage the cron's audit output, bundling it into their commit
3. **If another developer pulls and runs `git status`:** they see the cron's
   uncommitted changes mixed with their own work

### Can Git Auto-Resolve?

Not applicable — the cron doesn't create commits, so there's no git-level
conflict. The problem is **dirty working tree contamination**.

### When It Gets Worse: Cron With Auto-Commit

If someone adds auto-commit to the cron job:

```bash
# Dangerous cron:
0 2 * * * cd /path/to/hub && projectman sync && projectman audit \
  && git add -A && git commit -m "nightly sync" && git push
```

Now the cron creates a commit that:
- Updates all submodule refs to their latest remote commits
- Includes updated PM indexes, dashboards, and audit output
- Pushes to the hub's `main`

**Race condition with Dev A:**

```
Time   Dev A                              Cron (auto-commit)
────── ──────────────────────────────────  ──────────────────────────────────
01:59  git commit -m "api: late fix"       (not started yet)
02:00  git push   ← succeeds
02:00                                      projectman sync
                                           → pulls Dev A's api change ✓
                                           git add -A   ← stages ALL refs
                                           git commit -m "nightly sync"
                                           git push   ← succeeds
02:05  # More api changes...
       git add . && git commit && git push (in submodule)
       cd hub-root
       git add projects/api
       git commit -m "api: follow-up"
       git push   ← REJECTED
02:06  git pull --rebase
       # Conflict: both modified projects/api
       # Cron commit staged ALL refs, including api at T2:00
       # Dev A's commit advances api further
```

The cron's `git add -A` staged every submodule ref, creating a commit that
touches all of them. Any developer who subsequently pushes a submodule update
will conflict with the cron's commit on that submodule path.

### Recovery Path

**Standard cron (no auto-commit):**

```bash
# Clean up the dirty working tree left by cron
cd hub-root
git checkout -- .project/DRIFT.md .project/index.yaml  # Discard cron's changes
# Or, if you want to keep them:
git add .project/ && git commit -m "include nightly audit"
```

**Auto-commit cron (avoid this pattern):**

No clean recovery — every developer will hit conflicts until the next cron
cycle. The cron should either:
1. Not commit at all (recommended — leave audit output as dirty files)
2. Only commit `.project/` audit files, never submodule refs:
   ```bash
   git add .project/DRIFT.md .project/dashboards/
   git commit -m "nightly audit" && git push
   ```

### Data Loss Risk: LOW (standard cron) / MEDIUM (auto-commit cron)

- **Standard cron:** No data loss. Only risk is dirty tree contamination
  causing accidental staging of audit files
- **Auto-commit cron:** Submodule ref conflicts for every developer who pushes
  after the cron. No data is lost, but the conflict resolution is tedious and
  error-prone

## The Hub-on-Main Tension

### The Fundamental Problem

The hub repository stays on `main` because project management data should
never be branched:

- Stories, tasks, and epics represent the **current state of work** — forking
  PM data into feature branches creates divergent truths about project status
- Dashboards, indexes, and audit reports are hub-wide — they don't belong to
  any single feature
- The `.project/` directory is the single source of truth for the entire
  organization's PM state

But this means **all submodule ref updates compete on one branch**. Every
developer who touches any submodule must push to `main`. There is no isolation
between concurrent work streams.

### Why Branching the Hub Doesn't Work

```
main:     A ─── B ─── C ─── D ─── E
                 \                 ↑
feature/x:        F ─── G ─── H ──┘ (merge)
```

If Dev A branches the hub to work on `projects/api`:

1. **PM data diverges.** Dev A creates stories and tasks on their branch.
   Dev B, working on `main`, creates different stories. When Dev A merges,
   story IDs may collide, indexes are inconsistent, and acceptance criteria
   may contradict each other.

2. **Submodule refs are branch-specific.** The hub's `main` still points at
   old submodule commits. Dev A's branch points at new ones. Other developers
   on `main` don't see Dev A's submodule advances. When Dev A merges, the
   submodule refs jump forward, possibly skipping intermediate commits that
   other developers depend on.

3. **Dashboards and indexes are stale.** The hub's `main` dashboards don't
   reflect Dev A's branch work. Dev A's branch dashboards don't reflect
   Dev B's `main` work. After merge, both are wrong until someone runs
   `pm_reindex`.

### Consequences for Multi-Developer Workflows

Because all work funnels through `main`:

| Property                  | Effect                                        |
|---------------------------|-----------------------------------------------|
| **No isolation**          | Every push affects every other developer       |
| **Serial pushes required**| Two developers cannot push the hub simultaneously; one will always be rejected |
| **Merge window is global**| The time between `git pull` and `git push` is a race — longer tasks have higher conflict probability |
| **PM data is always contested** | `index.yaml` changes on every PM mutation, guaranteeing conflicts when two developers do PM work concurrently |
| **No "safe" time to push**| Even off-hours pushes can conflict with cron jobs or developers in other timezones |

### Mitigations (Within Current Architecture)

These don't solve the fundamental tension but reduce conflict frequency:

1. **Selective staging** — Never use `git add -A` in the hub. Only stage
   the specific submodule refs and `.project/` files you changed.

2. **Small, frequent pushes** — Minimize the merge window by pushing
   immediately after committing. Don't batch multiple submodule updates into
   one hub commit.

3. **Regenerate, don't merge, `index.yaml`** — After resolving a hub merge
   conflict, run `pm_reindex` instead of hand-merging the index. Add the
   regenerated file to the merge commit.

4. **Coordinate submodule ownership** — Assign submodules to developers so
   that two people rarely update the same submodule ref. This doesn't
   eliminate hub conflicts (`.project/` still conflicts) but reduces the
   most dangerous ones (submodule ref overwrites).

5. **Cron should not auto-commit** — The nightly sync/audit should leave
   dirty files for a human to review and commit, not create commits that
   conflict with developer pushes.

## Summary: Conflict Resolution Matrix

| Scenario | Git Auto-Resolves? | Data Loss Risk | Recovery Difficulty |
|----------|--------------------|----------------|---------------------|
| 1. Different submodule refs | Yes (clean merge) | None | Trivial — `git pull --rebase && git push` |
| 2. Same submodule ref | No (conflict) | Medium — one ref dropped | Medium — requires submodule branch merge |
| 3a. Stale hub (selective staging) | Yes (clean merge) | None | Trivial |
| 3b. Stale hub (`git add -A`) | No (conflict) | High — ref silently reverted | Medium — must detect and re-advance ref |
| 4a. Cron (no auto-commit) | N/A (no conflict) | None | Trivial — discard or stage cron output |
| 4b. Cron (auto-commit) | No (conflict on all refs) | Medium — tedious conflicts | High — every developer affected |

### Common Thread

All multi-developer conflicts share the same structural cause: **the hub is a
single-branch, multi-writer repository where submodule refs are opaque commit
SHAs with no semantic merge strategy.** Git cannot merge two different
submodule SHAs — it can only pick one or require manual resolution. Combined
with the hub-on-main constraint (PM data cannot branch), this means every
concurrent push is a potential conflict.

The highest-risk pattern is **`git add -A` in the hub**, which silently stages
stale submodule refs and can revert other developers' work without any warning.
This is the single most important practice to eliminate.
